# Resource name

Resource name is a *short, plain-language description of what it does and what problem it solves*. Two more sentences if needed — establish *why this resource exists* and *when a reader would use it instead of an alternative*. Avoid restating the OpenAPI summary; this is conceptual context.

## Lifecycle and state model {#lifecycle}

One paragraph on how a resource of this kind comes into being, transitions, and gets destroyed. A short list of possible states helps if there are more than three. Link to related resources where state changes propagate (e.g. "creating a VM also creates a default network interface").

## Relationships to other resources {#relationships}

One or two paragraphs on what this resource interacts with — its parent, its children, the resources it depends on, the resources that depend on it. Use bullets if there are more than three relationships. Each relationship should link to the corresponding reference page.

## Billing {#billing}

If billing for this resource is anything other than completely obvious, document it here. State the unit (per second, per GB-month, per request), the rate (or where to find it), and what gets billed vs. what's included. Skip this section entirely if the resource doesn't bill independently.

## Limits and caveats {#limits}

Per-project quotas, regional restrictions, "you can't do X with this resource because Y" — anything that a reader actively trying to use this resource will want to know up front. One short paragraph per limit.

- [Relevant concept](/docs/conventions) — the broader concept this fits into.
